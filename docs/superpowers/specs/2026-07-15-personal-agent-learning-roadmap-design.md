# CDY Agent 渐进式学习与演进设计

## 背景

CDY Agent 是一个用于学习 AI Agent 开发的 Python 项目。仓库已经主动清空到基础项目配置，适合从零重建。学习者已理解 Tool Calling、上下文和 Agent Loop 的基本概念，希望通过结对实现深入掌握工程实践。

项目的长期目标是一个运行在本地、通过命令行交互的个人助理。它将逐步获得会话、本地工具、Skills、持久化记忆和可观测性等能力。

## 目标与非目标

### 目标

- 使用 Python、Typer 和 OpenAI Responses API 构建个人助理。
- 采用纵向切片方式演进，每个阶段都交付可运行、可测试的版本。
- 通过小步结对实现理解 Agent 的数据流、边界和工程取舍。
- 为文件和 Shell 等本地能力建立明确的安全边界。
- 保持模块职责清晰，使关键组件可以独立理解和测试。

### 第一轮演进的非目标

- 不接入 LangChain 等现成 Agent 框架。
- 不提前抽象多模型支持。
- 不引入工作流引擎、多 Agent、向量数据库或 Web UI。
- 不为了未来需求预先创建没有实际行为的模块空壳。

## 演进方式

采用纵向切片式路线。每一步增加一个完整、可观察的能力，而不是先搭建全部抽象。早期实现可以在后续需求出现时重构；每次重构都需要说明旧设计遇到的具体限制。

演进顺序为：

1. 工程骨架。
2. 最小模型调用。
3. 多轮会话。
4. Agent Tool Loop 与本地工具安全边界。
5. 笔记与 Todo 工具。
6. Skills 系统。
7. 持久化与记忆。
8. 工程化与进阶能力。

## 总体架构

最初的数据流保持为：

```text
用户输入 -> CLI -> Agent -> OpenAI Responses API -> 最终回复
```

后续按需扩展为：

```text
CLI -> Agent Loop -> OpenAI Responses API
             |
             +-> Tool Registry -> Local Tools
             |
             +-> Skills
             |
             +-> Conversation and Memory
```

主要边界如下：

- CLI 负责终端输入输出，不承载 Agent 决策逻辑。
- Agent 负责模型调用、工具调用循环和终止条件。
- OpenAI SDK 调用在需要测试隔离时进入独立模块，但不演化为通用多模型接口。
- 每个工具只承担一个明确职责，通过统一结构注册和执行。
- 会话与记忆独立于终端界面，使未来可以替换交互入口。

## 目标模块

下列结构是演进目标，不要求第一阶段一次性创建：

```text
src/cdy_agent/
├── cli.py
├── config.py
├── agent.py
├── openai_client.py
├── conversation.py
├── tools/
│   ├── base.py
│   ├── registry.py
│   └── ...
├── skills/

│   └── ...
```

模块只依赖更底层的接口。工具不知道 CLI 的存在，OpenAI 客户端不负责会话持久化，存储层不参与模型决策。

## 工具调用数据流

一次带工具调用的请求遵循以下流程：

1. CLI 将用户输入交给 Agent。
2. Agent 将指令、会话上下文和工具定义发送给 Responses API。
3. 模型返回普通文本时，Agent 结束本轮并将文本交给 CLI。
4. 模型请求工具时，Registry 校验工具名称和参数。
5. 工具在自身安全边界内执行并返回结构化结果。
6. Agent 将结果关联到原工具调用，再次请求模型。
7. 最终回复、最大工具轮数或不可恢复错误会终止本轮。

## 阶段与验收标准

### 1. 工程骨架

建立最小 README、`src` 包、Typer CLI 和基础测试。验收标准是命令帮助、测试与包构建均正常运行。API 配置在第二阶段随首次模型调用加入。

### 2. 最小模型调用

实现单轮 Responses API 调用，覆盖请求、响应和基本异常处理。CLI 可以回答一个问题，自动测试不依赖真实网络。

### 3. 多轮会话

实现交互式 REPL、上下文延续和退出命令，明确模型调用与会话状态的职责边界。

### 4. Agent Tool Loop 与本地工具安全边界

实现同时支持 Responses API 和 Chat Completions API 的 Tool Loop，覆盖模型选择工具、程序执行、结果回传和循环终止。本阶段已经交付受工作区边界保护的 UTF-8 文件读取与写入，以及使用参数数组、允许列表、超时和输出上限的受限 Shell；写入和 Shell 调用均需用户确认。

### 5. 笔记与 Todo 工具

在已交付的工具注册、确认和安全边界上加入笔记与 Todo 等面向个人助理的能力。每个有副作用的能力必须同时实现参数校验、安全限制和测试。

### 6. Skills 系统

从本地目录发现技能说明和工具，建立可扩展的注册机制，并明确 Skills 与普通函数工具的职责区别。

### 7. 持久化与记忆

先保存会话，再加入由用户明确控制的长期记忆、检索和遗忘。默认使用 SQLite，不在此阶段引入向量数据库。

### 8. 工程化与进阶能力

加入结构化日志、调用轨迹、Token 与费用统计、配置分层、流式输出和评估用例。完成后再单独设计规划器、MCP、多 Agent 或 Web UI。

## 错误处理

- 配置错误在 CLI 启动阶段给出清晰提示，不默认展示冗长堆栈。
- 认证失败直接终止；限流和超时等暂时错误只进行有限次数重试。
- 未知工具、无效参数和工具执行失败转换为结构化结果，使模型可以解释或调整方案。
- Agent 设置最大工具调用轮数，防止无限循环；达到限制时返回明确诊断。
- 内部异常保留可调试信息，但面向用户的输出避免泄漏敏感配置和实现细节。

## 安全边界

- 文件工具只访问显式配置的工作目录，并在操作前解析真实路径，阻止路径穿越。
- 写入、覆盖、删除和 Shell 操作按风险分级；高风险操作必须得到用户确认。
- Shell 执行采用参数化调用、允许列表和超时，不直接拼接不受控命令字符串。
- 日志不记录 API Key、完整环境变量或敏感文件内容。
- 长期记忆必须允许用户查看、修改和删除，默认不保存所有对话。
- 自动测试只使用临时目录和模拟客户端，不访问用户真实文件或网络。

## 测试策略

测试分为三层：

- 单元测试覆盖配置、会话状态、工具参数、路径限制和注册分发。
- Agent Loop 测试使用伪造 Responses API 响应，覆盖直接回答、单次工具调用、连续工具调用、错误和轮数超限。
- CLI 测试覆盖参数、输出和退出码，不连接真实网络。

真实 API 调用只作为手动验收步骤。每个阶段至少运行：

```powershell
uv run pytest
uv run cdy-agent --help
```

涉及工程骨架或打包配置的阶段还必须运行 `uv build`。

## 结对学习流程

每个增量采用固定节奏：

1. 解释本次要解决的问题和最小设计。
2. 编写一个表达预期行为的失败测试。
3. 实现刚好让测试通过的代码。
4. 运行测试和 CLI 验证。
5. 阅读关键代码并复盘设计取舍。
6. 提供一个小型扩展练习。
7. 由学习者独立完成，或继续结对实现。
8. 里程碑保持独立、可回退，并使用简短明确的提交说明。

## 第一阶段边界

第一阶段只恢复最小工程骨架：README、包初始化、Typer CLI 和相应测试。它不调用 OpenAI API，不创建配置模块、工具、Skills、会话或记忆模块。完成标准是安装后的命令入口有效、帮助信息正确、测试通过且源码包与 wheel 可以成功构建。
